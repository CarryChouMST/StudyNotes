# 大模型与mcp

- mcp的调用正确性不受第三方保证

- 大模型与mcp的交互，一般是大模型组织输入驱动mcp响应；mcp响应结果与直接嵌入到大模型的输入中，这里面有个要注意的是，**某些工具返回的数据量非常大的情况下，系统为例性能和体验会在嵌入时直接截断**，==因此在执行一些大型工程任务时，最好一点一点的调用工具执行。==

- 大模型有点像大脑，只能基于输入提供**无目的**输出，不能与外界产生交互行为。mcp是外界对象提供的交互协议，此时需要增加一个客户端，让客户端基于该协议去执行交互。大模型与客户端+mcp二者相互协作就能产生影响，因此需要有个对象对接大模型和客户端，将大模型的**无目的输出**给到客户端，而协调这个协作的对象就是agent。而langchain可以更显式的自己协调协作方式，以对抗比如mcp信息错误等情况。

- api使用本质是不同的供应商有不同的协议，比如OpenAi, Google, Anthropic等，然后每个供应商有自己部署的模型，通过供应商制定的协议格式（包括url, 模型名称，apikey等）可以以api等方式访问模型，获取输入输出。一些模型提供商提供了兼容 OpenAI [Chat Completions API](https://platform.openai.com/docs/api-reference/chat) 的端点，在这种情况下，你可以通过设置自定义 `base_url`，使用 [`ChatOpenAI`](https://langchain-doc.cn/v1/python/integrations/chat/openai) 连接这些端点。

- MCP本质上与大模型一样，都是可以使用url访问的工具。

- MCP 支持用于客户端-服务器通信的不同传输机制：

  - **stdio**：客户端将服务器作为子进程启动，并通过标准输入/输出进行通信。最适合本地工具和简单的设置。
  - **Streamable HTTP**：服务器作为独立进程运行，处理 HTTP 请求。支持远程连接和多个客户端。
  - **Server-Sent Events (SSE)**：Streamable HTTP 的一个变体，针对实时流式通信进行了优化。

- 具体来说，大模型或者市面上常用的CLI是一种客户端，而MCP是一种协议，需要“服务器”实现该协议才能被ChatGPT等客户端加载。（类似访问网页都用http协议等，一种数据格式对接方式）。因此如果自己写了一个脚本(本地服务器)期望能与大模型直接对接，python有Server.mcp.server库提供了对脚本对包装方式。然后在ChatGPT等大模型客户端上配置访问方式即可。

  ```json
  {
  "mcpServers": {
  	"playwright": {
  		"command": "python",
  		"args": ["playwright_mcp.py"]
  								}
  							}
  }
  ```

- MCP的访问协议中，如果是stdio一般是启动子进程访问**本地工具**；WebSocket一般用来访问远程工具服务。

- Streamable\_http是MCP中专门用于 **通过HTTP+SSE**来做“流式调用”的传输模式，不是stdio和WebSocket，就是基于HTTP的SSE流式通信层。这是因为MCP虽然默认是双向WebSocket，但有的MCP server(特别是云服务)只支持HTTP，因此若想支持streaming，需要SSE。

- 【API协议】目前有三种API协议：OpenAI/Anthropic(Claude)/Vertex AI，这三种协议在langchain中都支持ChatOpenAI, ChatAnthropic, ChatVertexAI。
  - **OpenAI 格式**是最流行的，很多服务（通义千问、DeepSeek、Moonshot 等）都提供了兼容 OpenAI 的接口
  - **Anthropic 格式**相对独立，需要专门的 SDK
  - **Vertex AI 格式**是 Google 独有的，需要 Google Cloud 环境
  
- mcp中的tool有name和description两个核心字段，前者是名字，后者是描述。

- 添加mcp，就是在agent的设置目录添加mcpServers项下的键值对，名字：(协议与url/命令)。

- mcp有两种与服务器连接的方式，一种是通过command方式，通过本地命令启动mcp服务，如果是从本地连接到远端，则需要安装mcp-remote工具；第二种是通过serverurl的方式，直接通过url连接到预部署到mcp服务断电，适用于生产环境中的稳定服务连接。

- mcp的服务会在启动时向主模型注册自己提供的工具和能力，工具的描述信息通过API接口传递。

  ```
  result = chain.invoke({"content": "xxx文本"})
  print(result)
  ```

- 基于langchain构建LLM时，提供的url仅为base URl, SDK后台会补齐endpoint(注脚)。

- 在做复杂prompt时，推荐jinja2

  注意jinja2需要安装库，另外与f-string不能混用，这意味着，所有变量都需要使用{{}}，而且不能有f的前缀

  ```python
  原理：Jinja2 解析器会区分 {{...}}（变量/表达式）和 {%...%}（语句），其余的大括号会被原样保留。
  优点：适合复杂提示词，不需要手动转义 JSON 的大括号。
  prompt = ChatPromptTemplate.from_messages([
      ("system", """请返回以下格式的JSON：
  {
      "is_workflow": true,
      "name": "test"
  }
  {% if show_hint %}
  注意：请确保JSON格式正确。
  {% endif %}
  """), 
      ("user", "{{input}}") # Jinja2 中变量使用 {{var}}
  ], template_format="jinja2") # 关键：指定格式
  
  然后在使用时format_messages
  prompt.format_messages()返回的是一个消息列表
  formatted_messages = prompt.format_messages(message_input=message)
  ```

  - langgraph在invoke后，返回的结果是一个字典，其键与state一致，一般至少会有`messages`字段，一般可以在`最后一个AImessage`中，以协议键访问所需数据
  - 只能在最初定义state中的字段，添加新的字段会被自动社区
  - chain对象中，invoke的传参是一个dict，必须包含prompt中所有的参数，制定内容。而graph对象中，invoke传参是一个初始的state对象。
  - 继承 `BaseTool` 的类，本质是“给 LangChain/LangGraph/Agent 调用的工具”。它的**标准使用方式**只有两种：要么让 agent 调用，要么你自己用 `invoke/ainvoke` 按 `args_schema` 传参调用。不要自己随意改 `invoke/ainvoke` 的参数签名。
  - LangGraph 常见做法是把不可序列化资源放到：比如要开一个浏览器之类的
    - 全局单例（模块级）
    - 或者注入到工具闭包/依赖容器
    - 或者用 `contextvars`/运行时对象

## 标准化事件流

- 事件的元数据可以放sessionid等内容。

- ```
  token: 增量更新
  updates: 节点级状态更新
  event 模式输出的是“运行过程中的事件”
  values 模式输出的是“图的当前状态（state）的完整快照”
  ```

- 一般用token+values，event不够，因为有的中间件比如todo是在图状态中增加一个结构化状态，不属于langgraph的回调事件。但是从状态中拿回调事件又比较麻烦，所以event负责回调事件

- 在`self.agent.astream(stream_mode="values")`中，会将每次`迭代`后的`Agent状态快照`算作一次`事件event`。需要注意的是，中间件不是独立的迭代，他是节点执行过程中的“拦截器”或“增强器”

- ```
  StreamMode = Literal['values', 'updates', 'checkpoints', 'tasks', 'debug', 'messages', 'custom']  // 注意没有event
  ```

- 可以业务上自己推送事件流，比如标题等。

  ```
  from langchain_core.callbacks import dispatch_custom_event
  # ✅ 可以在这里推送自定义事件，前端会通过 SSE 的 custom 通道收到
  await dispatch_custom_event("turn_start", {
  "turn_id": turn_id,
  "session_id": session_id,
  })
  ```

- 在多工具调用时，tool-end不能只取最后一个消息，可能会有好几个ToolMessage。
- 在多订阅事件流时，不同类型的流可能由于体积的关系，输出不同步，比如toolend时，TOKEN类型的输出也在夹杂着输出。

## 多线程/多进程

- python有多线程、多进程和asynio（异步)的并发模型
- 多线程适合I/O密集型任务(比如网络请求、文件读写等)，但不适合CPU密集型。
- 多进程适合CPU密集型任务(数据处理、模型推理)
- 异步是单线程内操作，阻塞协程。
- 线程是CPU调度的基本单位；进程是系统资源分配的基本单位。
- 一般写agent框架，因为IO密集+脚本并行，适合异步和多进程，异步下协程的调度是由asyncio事件循环自动管理的。我们不管携程调度，只处理任务调度，比如每小时清理会话，每天清理一遍新产生的bug。
- 请求来自用户、不同渠道（后台、用户对话)、不同会话。OpenClaw的好处是，使用网关作为唯一真相来源，可以管理多渠道，多来源命令。
- `asyncio.Semaphore` 和 `asyncio.Lock` 都是 Python 异步编程中用于**并发控制**的工具，但它们的用途和行为不同。lock是互斥锁，Semaphore信号量，表示最多允许几个协程一起访问。

## 会话管理

### 配置加载器

`Class ConfigLoader`支持从YAML/JSON/环境变量加载，支持配置验证和默认值，配置热重载，单例模式。

- 自用：环境变量；多会话，会话配置；多用户，用户配置
- 有一个.env文件，每次启动会将它加载到环境变量，这个是在本地的。

### 会话管理器

`SessionManager`创建/销毁/查找对话，会话持久话，超时和清理，以及上下文管理，上下文窗口：管理对话历史的token显示，会话恢复，支持从持久化存储恢复会话。

- 单用户就直接用SQLite做异步读写、内存缓存热点会话、单连接池；如果是多用户就需要上redis这些数据库。
- 会话管理器主要是根据使用规模选择会话存储方式，可以直接文件存储，可以用简单SQLite数据库。一般用SQLite其实也是文件存储，只是存储形式和查询形式封装为例数据库引擎逻辑中。会话存储属于线程内数据保存，使用checkpointer就行。
- 空文件时，后端收到消息并创建sessionid，然后前端在流式响应开始时，将后端创建的sessionid带到前端

### 消息队列

- 单用户的顺序执行，不需要消息队列；如果是单用户的多任务执行，执行，则可以使用asyncio.Queue的优先级队列，可选SQLite持久话，放置重启后消息丢失。

- langgraph的state管理中，对于state的合并行为，如果一个step(一个node或者多个并行的node)内，出现了增量字段，只有一个node时会自动替换，如果有多个值，同时没有提供reducer合并函数，会直接报错。【如果有reducer会走另一个update，会轮询reducer】

  ~~~
  # langgraph/channels/last_value.py
  def update(self, values: Sequence[Value]) -> bool:
      if len(values) == 0:
          return False
      if len(values) != 1:
          raise InvalidUpdateError(...)  # 超过1个值就报错
      
      self.value = values[-1]  # 刚好1个值才替换
      return True
  ~~~

- 关键就在 `self.tools` 这个属性名 —— 这是框架（deepagents）约定的魔法属性。当你把中间件加到主 Agent 时，框架会**自动把 `self.tools` 里的工具合并进主 Agent 的工具列表**。

# Skill

- skill包括skill.md, reference.md, scripts和assets，主要用于技能描述，参考知识，相关脚本和资产
- skill相比于subAgent好处在于上下文共享，但是对于复杂垂直领域，大量领域知识可能会污染主上下文，也可能被主上下文淹没。
- subAgent用于处理子任务复杂，中间过程繁琐，需要并行处理多个独立任务时。
- Skills 是应用程序，装在主系统里按需调用；SubAgent 是虚拟机，独立运行完再把结果交回来。**任务简单用应用，任务复杂开虚拟机。**
- 并行、隔离、特殊权限用subAgent，同一套上下文用skill。

# A2A

- 无论是A2A还是multi-Agent还是team Agent本质上都是多智能之间的协作，协作包括协作模式和通信方式，目前的协作模式有串行、辐射，通信方式一般是直接通信，claudecode中team在于协调：共享任务列表，通信：双向通信，适合：subAgents适合只有结果的专注任务，而teams适合需要讨论和协作的复杂工作。

## Agent服务构建的坑

- 运行长时工具时，可能会有长时间没有token往请求客户端发，导致断连。
- 除了**空闲超时**之外，还存在**请求超时**。整个请求链路是**浏览器->SLB—>FastAPI服务->服务器返回的连接类型(http/SSE或WebSocket)**
  - SLB对不同协议的超时策略不一样，对于HTTP/SSE，受`request_timeout`限制(最大180s/900s)，到时间就断
